## 1、简答
 1. <font color="red">Difference and relationship of Software Requirements,Quality Attributes and Architectural Significant Requirements</font>
   ***G:*** 软件需求：包含功能需求（系统要干什么事）和非功能需求（系统要做的多好）；质量属性：属于非功能需求，是衡量系统运营和维护表现的指标（如可用性、性能、可修改性、安全性）；ASR（架构显要需求）：是需求的一个子集，并非所有需求都影响架构，只有对架构设计产生决定性影响或跨越多个模块的需求才是ASR

    关系：软件需求包含质量属性；质量属性经过筛选、具备架构冲击力后，成为 ASR。架构设计的核心目标就是为了满足 ASR。


 2. <font color="red">6个通用架构策略</font>
    ***G***：
    可用(可用性 - Availability): Ping/Echo, Heartbeat（心跳检测故障）。

    可修(可修改性 - Modifiability): Encapsulate (封装), Use an Intermediary (引入中介如消息队列)。

    可测 (可测试性 - Testability): Separate Interface from Implementation (接口与实现分离，便于 Mock)。

    迭 (迭代/易用性 - Usability): Support User Initiative (支持用户主动撤销/重试)。

    安 (二/安全性 - Security): Authenticate Actors (身份验证), Limit Access (权限控制)。

    分 (分/性能 - Performance): Manage Resources (利用缓存/多线程并发处理)。
 3. <font color="red">如何建模质量属性，并画出可用性和可修改性图</font>
   刺激相应图：环境；刺激源 ;刺激；工件；响应；响应度量


    “可用性的这个图就是小人儿那儿刺激源是心跳监测器，然后刺激是检测到服务未应答，或者服务停机环境是系统掌工作室，然后工件儿是系统进程，然后响应是降级处理切换备用节点儿，或者是通知运维人员。然后响应度量是系统30秒内恢复正常，用户感机感知的停机时间为0。”
   
 4. <font color="red">架构Pattern和Tactic的区别和联系</font>
    策略比模式更简单，仅有单一的结构或机制来应对单一的架构驱动 

    策略是创建架构设计的重要组成块。 

    模式通常将多个设计决策组合到一个包中。

    模式和策略共同构成了软件设计师的主要工具。

    大多数模式包含集中不同的策略，这些策略可能有共同的目的或者经常被选择来实现不同的质量属性。
 5. <font color="red">架构设计主要Process以及Input/Output</font>
   * 明确ASR（架构攸关需求）: 输入利益相关者的期望，商业目标等，输出排好优先级的质量属性场景
   * 架构设计：输入：排好优先级的质量属性场景、模式和策略，需求和约束；输出：候选视图的草图
   * 文档化：输入：候选视图的草图，利益相关者的审查，，输出选定且整合过的视图和补充文档
   * 架构评估：输入拍好先后顺序的质量属性，利益攸关者审查，整合过的视图和补充文档，输出：再次整合过的视图和补充文档，不断迭代
  
 6. <font color="red">相比 OOD，引入 DDD 中 Aggregate Pattern 的价值</font>
   ***G***：
   传统 OOD 的问题: 传统面向对象设计中，对象之间存在错综复杂的网状关联（如 Order 关联 LineItem，LineItem 关联 Product）。在并发高、分布式环境下，很难维护业务事务边界和数据一致性。

   Aggregate (聚合) 的价值:
   
   划分显式边界 (Explicit Boundary): 将一组紧密相关的领域对象（实体和值对象）组合在一起，作为一个整体进行数据修改。

   保护业务不变性 (Invariants): 外部只能通过聚合根 (Aggregate Root) 来访问和修改聚合内部的对象，确保所有业务规则在边界内强制生效。

   优化并发与锁定: 锁定的粒度从数据库表或独立的无数小对象，变成了以“聚合根”为单位，极大地提升了并发性能和分布式系统的可扩展性。

 8. <font color="red">微服务系统的 4 个 Architectural Element 及其 Roles</font>
    ***G***：
    API Gateway (API 网关): Role 是作为统一入口，路由外部请求，负责身份验证、负载均衡和限流。

    Service Registry / Discovery (服务注册与发现): 是动态维护所有微服务实例的 IP 和端口列表，允许服务间动态寻址。

    Microservice Instances (微服务独立实例):  是承载具体的单一业务领域逻辑，拥有自己独立的数据库，独立部署。

    Centralized Configuration Server (配置中心) / Event Bus (事件总线):  是统一管理分布式环境下的配置 / 或实现微服务间的异步解耦通信。
 9.  <font color="red">Event-Driven 架构风格的 Basic Idea 与质量属性 Trade-off</font>
   ***G***：
   Basic Idea: 系统由事件的产生、检测、消费和反应来驱动。组件（事件生产者）发出事件通知（Event），不关心谁去消费；一个或多个订阅者（事件消费者）异步监听并处理这些事件。

   Trade-off (权衡):

   Pros (优点/提升的质量属性): Modifiability / Scalability / Decoupling (高可修改性与可扩展性)。生产者和消费者完全解耦，增加新功能只需订阅事件，无需改动原系统。

   Cons (缺点/降低的质量属性): Testability / Debuggability / Consistency (低可测试性与一致性)。由于是异步和非确定性的，追踪消息流非常困难，且只能保证最终一致性而非强一致性。
 10. <font color="red">企业架构中的4A及其Relationship</font>
   ***G***：
   BA (Business Architecture, 业务架构): 业务战略、流程、组织架构。

   DA (Data Architecture, 数据架构): 数据资产管理、数据流、数据库模型。

   AA (Application Architecture, 应用架构): 系统的划分、应用之间的交互与功能实现。

   TA (Technology Architecture, 技术架构): 基础设施、硬件、网络、中间件等。

   BA 是源头和驱动力，决定了系统需要什么应用（AA）和需要处理什么数据（DA）。

   DA 和 AA 互为表里，应用架构实现业务逻辑并产生/消费数据，数据架构为应用提供数据支撑。

   TA 是底层基座，为 AA 和 DA 的运行提供物理硬件和技术中间件的支撑。
